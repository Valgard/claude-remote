# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A thin shell wrapper that makes interactive Claude Code sessions remotely attachable from another device (tested with an iPad over Blink Shell on the LAN). The only owned code is two stateless shell scripts and a shared library (plus `cr-sign-tmux`, a one-off macOS maintenance helper); persistence, transport, auth, and metadata are delegated to existing tools (tmux, sshd, abtop, fzf). There is no daemon of our own.

Read `docs/specs/2026-06-15-claude-remote-design.md` for the full design rationale before making non-trivial changes.

## Commands

```bash
make test          # run the bats suite (tests/)
make lint          # shellcheck the bins, the lib, and install.sh
make fmt           # shfmt -w -i 2 -ci  (2-space indent, switch-case indented)
make fmt-check     # shfmt diff check (CI-style, no write)
make sign-tmux     # rebuild + ad-hoc sign tmux for macOS Local Network privacy

bats tests/join.bats              # run a single test file
bats -f "marks attachable" tests/ # run tests matching a name pattern
```

`shfmt -i 2 -ci` is the canonical format; every change to `bin/`, `lib/`, or `install.sh` must pass `make fmt-check` and `make lint` (the repo runs clean under shellcheck — keep it that way, with targeted `# shellcheck disable=` comments only where intentional, e.g. `SC2086` word-splitting on `$CR_TMUX`).

## Architecture

**Three source files, one library:**

- `lib/claude-remote-lib.sh` — every piece of real logic lives here as a `cr_*` function. Sourced by both bins **and** the tests, so logic is unit-testable without invoking the scripts end-to-end.
- `bin/claude-remote` — launch wrapper: parses `-l/--label`, `--no-attach`, `--`, then calls `cr_session_name` + `cr_launch`. `set -euo pipefail`.
- `bin/claude-remote-pick` — the picker: a redraw loop calling `cr_menu_lines` and one of the pick functions. Also the SSH forced-command entry point, and the `--ensure-anchor` entry point used by the install.sh LaunchAgent (see Keychain anchor below). `set -uo pipefail` (no `-e`: the attach/redraw loop must survive non-zero exits).

**The pid join (the core idea).** `cr_launch` runs `claude` *directly* as the tmux exec target (`tmux new-session -d -- claude "$@"`), so `pane_pid == claude pid`. That shared pid is the join key: `cr_abtop_sessions` emits one TSV row per Claude session keyed by pid, `cr_pane_map` emits `pane_pid<TAB>session_name` for every tmux pane, and `cr_join` matches them — rows with a pane become attachable `S` rows, the rest are counted into a single `N` (running but not under claude-remote, so not attachable). Launching via the exec form also bypasses the user's interactive `claude` zsh function (equivalent to `command claude`), avoiding infinite recursion.

**The display pipeline** (all stdin/stdout TSV, composable and individually testable):

```
cr_abtop_sessions ─┐
                   ├─> cr_join ─> S/N rows ─┬─> cr_format_rows ─> "session<TAB>display"
cr_pane_map ───────┘                        └─> cr_footnote     ─> "(N weitere …)" on stderr
                                  cr_menu_lines wraps the above; falls back to a raw
                                  `tmux list-sessions` when abtop is unavailable.
```

The first column of every menu line is the tmux session name (the attach key); the rest is human-facing display. `cr_format_rows` pads on *visible* length so ANSI colour (gated by `CR_COLOR=1`) doesn't break column alignment.

**Picker selection** returns a token, not a raw choice: a session name to attach, or `__NEW__` / `__QUIT__` / `__RELOAD__` / `__NONE__`, or the compound `__KILL__<TAB>session` (kill that session). `cr_pick_fzf` (used when `fzf` exists and stdout is a tty) and `cr_pick_numbered` (fallback) both honour the same token contract, so the loop in `bin/claude-remote-pick` is UI-agnostic. The kill token is compound because — unlike the parameterless tokens — it must carry the *target*: fzf's `--expect=ctrl-r,ctrl-x` reports `Ctrl-X` plus the highlighted line in one go, and the numbered menu parses a `k<n>` choice. Both guard against a non-session target (the synthetic `＋ neue Session` entry / out-of-range `n`) by returning `__NONE__`. The actual kill is `cr_kill_session` (`tmux kill-session` + drop the post-exit capture file; refuses `$CR_ANCHOR` and an empty target); the loop wraps it in a `[j/N]` confirm (default No), read from fd 0 like the `mkdir` confirm.

**Attach semantics.** Attaching is *never* `exec`'d — the caller must regain control afterwards to drain Claude's post-exit output (see below). `cr_attach` is the bare `$CR_TMUX attach`; `cr_attach_and_drain <session> <wait>` wraps it with the drain + an optional keypress pause; `cr_launch` (new session) and `cr_reattach` (running session) both delegate to it. The picker's new-session call still runs in a `( … )` subshell so the `cd` stays local. `<wait>` is the one bit that differs by caller: the picker passes `1` (pause so the drained output survives the next full-screen redraw — fzf uses the alternate screen), the wrapper passes `0` (its output just stays in the scrollback above the returning prompt). The drain lives **only** in `cr_attach_and_drain`, never in `cr_launch`/`cr_configure_exit_capture`, so both picker paths converge on a single drain call that decides whether to pause — were `cr_launch` to drain, its new-session subshell would consume the capture file before the picker could tell whether there was output. The pause reads its keypress from `/dev/tty`, **not** fd 0: in the picker loop a read from fd 0 returns immediately after `tmux attach` returns over SSH (so it would not actually wait), whereas `/dev/tty` blocks for a real key.

**Post-exit output capture (the `[exited]` fix).** Claude Code runs as a full-screen TUI in the terminal's *alternate* screen buffer, so its scrollback is worthless — but on exit it leaves the alt-screen and prints a couple of lines (e.g. the session id) to the *primary* screen, then the process ends. Normally tmux destroys the pane/window/session in the same instant and the attached client only sees `[exited]`, losing those lines. `cr_configure_exit_capture <session>` arms the session; it runs **at launch and on every re-attach**, so a session started by older code — or a bare `tmux new-session -- claude` — gets armed the first time you re-attach through the picker (except `_cr_anchor`, which `cr_menu_lines` hides, so it is never a re-attach target and never armed). It is idempotent by construction: the first `set-hook` (no `-a`) *replaces* the whole `pane-died` hook and the three `set-hook -a` re-append, so re-arming yields the identical 4-command hook with no accumulation. It sets `remain-on-exit on` (window option) so the dead pane is *held* — which both preserves its grid and fires the `pane-died` hook — and installs that 4-command hook (pure tmux commands, no `run-shell 'tmux …'`, so they hit *this* server's socket): `capture-pane -b <buf> -S - -E -` → `save-buffer -b <buf> <file>` → `delete-buffer` → `kill-session`. The `<buf>`/`<file>`/`<session>` values are baked into the hook strings by our shell at set-time (not via tmux `#{…}` runtime formats), so each stays per-session unique and free of expansion surprises. The `-S - -E -` (full history) is load-bearing: the dead-pane banner repaint pushes the top line(s) out of the *visible* grid, so a visible-only capture would lose them. `kill-session` ends the session itself, so there is no zombie (abtop has no dead pid → the picker is unaffected). The join invariant is untouched — the pane's exec target stays `claude`; wrapping it in a shell to hold the pane open (`claude "$@"; exec $SHELL`) is **forbidden**, as that would make `pane_pid` the shell's pid and break the abtop↔tmux `cr_join` — which is exactly why `remain-on-exit` (orthogonal to the exec target) is the chosen lever. `cr_drain_exit_output <session>` then prints *that session's* file — session-specific on purpose, so a stale or another session's leftover capture is never surfaced and a plain detach (the session is still alive, so it has no capture) correctly shows nothing — dropping tmux's `Pane is dead (status …)` banner and the blank padding (no header), and returns 0 iff something was shown. It deliberately does **not** delete the file: a dying session can have several clients attached at once (e.g. the iTerm window it was launched in *and* a re-attached iPad), and each returns from `tmux attach` and drains — deleting on read would let whichever client wins the race consume it and leave the others (notably the picker, which wants to pause) with nothing. Cleanup is therefore out of band: `cr_reap_exit_files` (called at the picker loop top and once by the wrapper) deletes capture files older than `CR_EXIT_TTL_MIN` (default 1 minute) — by then every attached client has long since read it and the session is dead, so it never races a reader.

**Path-prompt editing.** The `＋ neue Session` directory prompt reads with `read -e` (readline) so the cursor keys edit the line over SSH (e.g. from an iPad); a plain `read` would leak raw `ESC[`-prefixed arrow-key bytes into the buffer instead. `read -e` engages readline only on a tty, so piped/test input still behaves like a plain read. Keep the `-e` — the full rationale lives in the `bin/claude-remote-pick` comment.

**Keychain anchor (macOS).** A tmux server's launchd bootstrap namespace is fixed at birth and inherited by every pane. If the picker births the server from inside the iPad's SSH forced command (no server running yet), it lands in the `Background` domain, where Claude Code's login-keychain write (OAuth token refresh) fails with `errSecInteractionNotAllowed (-25308)` — there is no SecurityAgent route outside the GUI (`Aqua`) session. `cr_ensure_anchor` fixes this by birthing a hidden holding session (`CR_ANCHOR`, default `_cr_anchor`) *only when no server runs yet* (a no-op otherwise — never disturb a server it didn't birth). `install.sh` installs a per-user `LaunchAgent` (`de.valgard.claude-remote-anchor`, `LimitLoadToSessionType=Aqua`, `RunAtLoad` + `StartInterval=CR_ANCHOR_INTERVAL`, default 60s) that runs `claude-remote-pick --ensure-anchor` at GUI login and periodically (self-healing: re-establishes an `Aqua` server within one interval; the anchor session is hidden from the picker via `cr_menu_lines`). Only the *server birth* is anchored, never the `-- claude` pane command, so `pane_pid == claude pid` stays intact. A long-lived `CLAUDE_CODE_OAUTH_TOKEN` (which would bypass the keychain) was rejected because the user switches Team↔Max20x plans by re-login: Claude Code's macOS credential store is the login keychain (last in an auth-precedence chain — earlier tiers like `ANTHROPIC_API_KEY` / `CLAUDE_CODE_OAUTH_TOKEN` bypass it; there is no flag to disable the keychain on macOS), so native keychain OAuth must stay. Only meaningful while logged into the Mac GUI (the keychain is unreachable otherwise).

**Local Network privacy (macOS).** macOS gates LAN access per app via the Local Network privacy permission, attributed to the *responsible process* — for picker sessions that is the tmux server. Homebrew's tmux carries no `Info.plist`, so macOS treats it as unidentified and silently blocks LAN connections from picker-born sessions (e.g. `git push` to a remote on the LAN, "no route to host") while public internet still works. The permission pane has no "+" to add a CLI tool, and Local Network state is not managed by `tccutil`/MDM, so it can't be granted the usual ways. `bin/cr-sign-tmux` (via `make sign-tmux`) rebuilds tmux from source with an embedded `Info.plist` (`CFBundleIdentifier` + `NSLocalNetworkUsageDescription`) and ad-hoc signs it — turning tmux into an identified app macOS *can* grant. It smoke-tests the new binary on an isolated socket and backs up the original (`<tmux>.cr-orig`) before installing. Activation needs `tmux kill-server` (reloads the binary off disk) plus a one-time "Allow" on the macOS prompt; re-run after `brew upgrade tmux` (Homebrew overwrites the patched binary). `install.sh` never builds/replaces tmux itself — it only prints a hint (via `cr-sign-tmux --check`, read-only) when the installed tmux is unpatched. **Known issue:** on some setups the one-time grant prompt never fires after `sign-tmux` and tmux never lists under *Local Network*, so the permission can't be completed and LAN stays blocked despite the embedded `Info.plist` (suspected: ad-hoc signature has no stable TeamID, and/or the server is born outside a registrable GUI-app context via the LaunchAgent / SSH forced command). Apple-signed binaries are LNP-exempt regardless of the responsible process, so the robust, session-preserving fallback for git-to-a-LAN-remote is SSH transport instead of HTTP: `git config --global url."ssh://git@<host>:<ssh-port>/".insteadOf "http://<host>:<http-port>/"` — no `kill-server` needed. Diagnosis: `nc -z <host> <port>` (also Apple-signed, exempt) succeeds from a session where Homebrew `curl`/`git` get `no route to host`, confirming it's LNP and not the network.

## Conventions that matter

- **Env seams for testability:** all external-tool access goes through `CR_TMUX` (default `tmux`), `CR_ABTOP` (default `abtop`), `CR_SSH_PORT` (default `22`), `CR_ANCHOR` (default `_cr_anchor`, the keychain-anchor holding session), `CR_EXIT_DIR` (default `<tmpdir>/claude-remote-exit` — where `pane-died` hooks drop captured post-exit output for the wrapper/picker to drain; tests point it at an isolated dir). `CR_NEW_DIR` (default `~/Projects`) is the starting directory the picker's `＋ neue Session` entry offers when you press Enter at the directory prompt; it is also the base for the directory prompt's bare-name shortcut. The prompt's input is resolved by `cr_resolve_new_dir` (pure, unit-tested): a bare name with no `/` and no leading `~`/`$` (e.g. `myproject`) becomes `$CR_NEW_DIR/<name>` — ergonomic on a remote keyboard where `~` and long absolute paths are awkward — while explicit paths, `~/…`, and `$HOME/…` are taken as typed. The function tags the result `bare` or `path`; a missing **bare-name** dir is offered for creation (`Anlegen? [j/N]`, default no), a missing explicit path is still rejected. The only non-pure part (the confirm prompt + `mkdir -p`) stays inline in `bin/claude-remote-pick`. Never call `tmux`/`abtop` directly in `lib/` — go through the variable. Tests override these to point at an isolated `tmux -L <socket>` server and stub binaries (`tests/fixtures/abtop-stub`, `tests/fixtures/fake-claude`), so the suite never touches the user's real sessions.
- **Symlink-safe sourcing:** both bins resolve `BASH_SOURCE` through symlinks before sourcing the lib, because `install.sh` symlinks them into `~/.local/bin` while the lib stays next to the real script. Preserve this loop when editing the script headers.
- **Stripped PATH:** `cr_augment_path` appends Homebrew / local bin dirs to `PATH` so `abtop`/`tmux`/`jq`/`fzf`/`claude` resolve under an SSH forced command (whose PATH is minimal). It *appends* (never prepends) so existing PATH entries keep priority. Among the dirs it adds, `~/.local/bin` / `~/local/bin` come **before** `/opt/homebrew/bin` — a binary pinned into the user-local bin (e.g. a specific native `claude`) then wins over a Homebrew copy. This is load-bearing: a tmux pane inherits the launching client's PATH, so without this order a launch from an unstripped Mac shell (where `~/.local/bin` precedes Homebrew) and one from the stripped SSH picker would resolve `claude` to *different* binaries.
- **Stripped locale:** the same stripped env leaves `LANG`/`LC_*` empty, so a tmux server born there (and every Claude pane under it) runs on the C/US-ASCII locale — tmux still *renders* UTF-8 via the client's utf8 flag, but pasted multibyte UTF-8 is mangled by the ASCII input layer. `cr_ensure_utf8_locale` pins a UTF-8 locale **before any server is born** (both bins call it right after `cr_augment_path`; in `claude-remote-pick` ahead of the `--list`/`--ensure-anchor` branches so the anchor server birth is covered too). It is a no-op when the effective locale (`LC_ALL`>`LC_CTYPE`>`LANG`) is already UTF-8, and only sets a locale `locale -a` actually lists. `CR_LOCALE` (default `de_DE.UTF-8`, then `en_US.UTF-8`, then `C.UTF-8`) overrides the target.
- **Idempotent installer:** `install.sh` must be re-runnable after a pull. Line appends to `~/.tmux.conf` go through `cr_ensure_line` (present-check + newline-safe append) — never a bare `>>`.
- **User-facing strings are German; code, comments, and docs are English.** Match the existing mix (menu prompts, footnotes, and warnings in German; everything else English).
- **bash 3.2 compatibility:** macOS ships bash 3.2. Avoid injecting empty array elements under `set -u` (see the `pick_args` guard in `bin/claude-remote-pick`).

## Tests

bats-core suite under `tests/`, one file per concern (`join`, `render`, `pane_map`, `name`, `fallback`, `pick`, `launch`, `install`, `path`, `resolve`, `locale`, `sshd`, `abtop_parse`, `anchor`, `exit_output`, `kill`). Each sources `tests/helpers.bash` via `load helpers`:

- `cr_setup` / `cr_teardown` — isolated tmux server (unique `-L` socket per test), stub `abtop` and `claude` on PATH.
- `cr_make_session <name>` — start a fake-claude session and echo its `pane_pid`.
- The `fake-claude` stub has an exit mode (`FAKE_CLAUDE_EXIT_LINES`, optional `FAKE_CLAUDE_EXIT_DELAY`): instead of sleeping forever it waits, prints the given lines, and exits — used by `exit_output.bats` to drive the `pane-died` capture end to end (it inherits the env via the tmux server, so export before the first tmux call).

Most tests source a single `cr_*` function and feed it TSV on stdin, asserting on stdout — match that style (small, function-level, no real network or real Claude) when adding coverage.
